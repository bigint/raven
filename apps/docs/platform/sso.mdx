---
title: "Single Sign-On (SSO)"
description: "Configure SAML or OIDC authentication for your organization."
---

Raven supports enterprise single sign-on (SSO) via **SAML** and **OpenID Connect (OIDC)** for centralized identity management. SSO lets your team authenticate through your existing identity provider instead of managing separate Raven passwords.

<Note>
  SSO configuration is available on **Enterprise** plans. GitHub and Google OAuth are available on all plans.
</Note>

## Supported Protocols

| Protocol | Description | Best For |
|----------|-------------|----------|
| **OIDC** | OpenID Connect -- Modern, token-based SSO | Most organizations |
| **SAML** | Security Assertion Markup Language | Legacy enterprise environments |

## OIDC Presets

Raven includes presets for popular OIDC providers with pre-configured scopes and attribute mappings:

| Provider | Issuer URL | Auto-Configured |
|----------|------------|-----------------|
| **Google** | `https://accounts.google.com` | Scopes, attribute mapping |
| **Microsoft** | `https://login.microsoftonline.com/common/v2.0` | Scopes, attribute mapping |
| **Okta** | Your Okta domain | Scopes, attribute mapping |
| **Auth0** | Your Auth0 domain | Scopes, attribute mapping |
| **Keycloak** | Your Keycloak realm URL | Scopes, attribute mapping |

All presets use the standard scopes: `openid`, `profile`, `email`.

## Configuring OIDC

<Steps>
  <Step title="Navigate to SSO Settings">
    Go to **Settings > SSO** in the dashboard.
  </Step>
  <Step title="Select OIDC">
    Choose OpenID Connect as the protocol.
  </Step>
  <Step title="Choose a Preset or Custom">
    Select a preset (Google, Microsoft, Okta, Auth0, Keycloak) or configure a custom OIDC provider.
  </Step>
  <Step title="Enter Configuration">
    Provide the required values from your identity provider.
  </Step>
  <Step title="Test the Connection">
    Verify SSO works by signing in with your identity provider.
  </Step>
</Steps>

### OIDC Configuration Fields

| Field | Required | Description |
|-------|----------|-------------|
| `clientId` | Yes | OAuth client ID from your IdP |
| `clientSecret` | Yes | OAuth client secret from your IdP |
| `issuerUrl` | Yes | OIDC issuer URL (used for discovery) |
| `authorizationUrl` | No | Override the authorization endpoint |
| `tokenUrl` | No | Override the token endpoint |
| `userInfoUrl` | No | Override the userinfo endpoint |
| `scopes` | No | OAuth scopes (defaults: `openid`, `profile`, `email`) |
| `callbackUrl` | Auto | ACS/callback URL (provided by Raven) |

### Attribute Mapping

Map your IdP's user attributes to Raven fields:

| Raven Field | Default Mapping | Description |
|-------------|-----------------|-------------|
| `email` | `email` | User's email address (required) |
| `name` | `name` | User's display name |
| `sub` | `sub` | Unique user identifier |

## Configuring SAML

<Steps>
  <Step title="Navigate to SSO Settings">
    Go to **Settings > SSO** in the dashboard.
  </Step>
  <Step title="Select SAML">
    Choose SAML as the protocol.
  </Step>
  <Step title="Enter IdP Configuration">
    Provide the SSO URL, issuer, and signing certificate from your identity provider.
  </Step>
  <Step title="Configure Your IdP">
    Copy the ACS URL and SP Entity ID from Raven into your IdP's configuration.
  </Step>
</Steps>

### SAML Configuration Fields

| Field | Required | Description |
|-------|----------|-------------|
| `entryPoint` | Yes | IdP SSO URL |
| `issuer` | Yes | SP entity ID |
| `certificate` | Yes | IdP signing certificate (PEM format) |
| `callbackUrl` | Auto | Assertion Consumer Service (ACS) URL |
| `signatureAlgorithm` | No | `sha256` (default) or `sha512` |
| `nameIdFormat` | No | NameID format for SAML assertions |

### SAML Attribute Mapping

| Raven Field | Default Mapping | Description |
|-------------|-----------------|-------------|
| `email` | `email` | User's email from SAML assertion |
| `name` | `name` | User's display name |
| `groups` | `groups` | Group membership (optional) |

## JIT Provisioning

When **Just-in-Time (JIT) provisioning** is enabled, users are automatically created in Raven when they sign in via SSO for the first time. No manual invitation is required.

| Setting | Description |
|---------|-------------|
| **JIT Provisioning** | Auto-create users on first SSO login |
| **Default Role** | Role assigned to JIT-provisioned users (e.g., `developer`) |

<Info>
  JIT-provisioned users inherit the default role you configure. You can change their role after provisioning from the Team page.
</Info>

## Enforcing SSO

When SSO enforcement is enabled:

- All organization members must authenticate via SSO
- Email/password login is disabled for the organization
- GitHub and Google OAuth are disabled for the organization
- New members can only join through the SSO flow

This ensures all authentication goes through your identity provider, giving your IT team centralized control over access.

## Allowed Domains

Restrict SSO to specific email domains:

```json
{
  "allowedDomains": ["acme.com", "acme.io"]
}
```

When configured, only users with email addresses matching the allowed domains can authenticate via SSO. This prevents users outside your organization from gaining access even if they have valid IdP credentials.

## Built-in OAuth

GitHub and Google OAuth are available on all plans without any SSO configuration:

| Provider | Required Config |
|----------|----------------|
| **GitHub** | `GITHUB_CLIENT_ID`, `GITHUB_CLIENT_SECRET` |
| **Google** | `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET` |

These are configured at the platform level, not per-organization.

## Session Management

Raven tracks active sessions with:

- **IP address** -- Where the session was created
- **User agent** -- Browser and device information
- **Created at** -- When the session started

Users can view and revoke their active sessions from their profile settings.
