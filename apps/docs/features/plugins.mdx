Plugins are hooks that run at specific points in the request pipeline, allowing you to transform requests, modify responses, handle errors, and filter streaming content. Raven includes built-in plugins for common tasks and supports custom plugins for organization-specific logic.

## Plugin Hooks

Plugins attach to one of four hook points in the request lifecycle.

| Hook | When It Runs | Use Case |
|------|-------------|----------|
| `pre_request` | Before the request is sent to the provider | Modify prompts, inject system messages, validate content |
| `post_response` | After the response is received from the provider | Transform output, redact PII, fix malformed JSON |
| `on_error` | When the provider returns an error | Custom error handling, fallback logic, alerting |
| `on_stream` | During streaming responses | Real-time content filtering, progress tracking |

## Built-in Plugins

Raven ships with two built-in plugins that can be enabled for any organization.

### Response Healing

**Hook:** `post_response`

Automatically fixes malformed JSON in LLM responses. When a model returns invalid JSON, this plugin attempts to repair it by:

1. Removing trailing commas before `]` or `}`
2. Adding missing closing braces `}` and brackets `]`
3. Validating the repaired JSON before returning it

```json
// Before healing
{"name": "Alice", "items": ["a", "b",}

// After healing
{"name": "Alice", "items": ["a", "b"]}
```

<Info>
  Response Healing only modifies the response if the repaired JSON is valid. If the JSON cannot be fixed, the original response is returned unchanged.
</Info>

### PII Redactor

**Hook:** `post_response`

Scans LLM responses for personally identifiable information and replaces it with redaction markers.

| PII Type | Replacement |
|----------|-------------|
| Email addresses | `[EMAIL REDACTED]` |
| Social Security Numbers | `[SSN REDACTED]` |
| Credit card numbers | `[CARD REDACTED]` |

```
// Before redaction
Contact john@example.com or call 555-123-4567. SSN: 123-45-6789.

// After redaction
Contact [EMAIL REDACTED] or call 555-123-4567. SSN: [SSN REDACTED].
```

<Note>
  The PII Redactor operates on responses from the provider. To block PII in incoming prompts, use [Guardrails](/features/guardrails) with PII detection.
</Note>

## Creating a Custom Plugin

### Via the Dashboard

<Steps>
  <Step title="Navigate to Plugins">
    Go to **Plugins** in the dashboard sidebar.
  </Step>
  <Step title="Click Create Plugin">
    Enter a name, description, and version number.
  </Step>
  <Step title="Select Hooks">
    Choose which hook points the plugin should attach to.
  </Step>
  <Step title="Add Configuration">
    Define the plugin's configuration options and any custom logic.
  </Step>
  <Step title="Enable the Plugin">
    Toggle the plugin to enabled. It starts processing requests immediately.
  </Step>
</Steps>

### Plugin Configuration

Each plugin has the following fields:

| Field | Description | Default |
|-------|-------------|---------|
| `name` | Plugin display name | Required |
| `description` | What the plugin does | Empty |
| `hooks` | Which hook points to attach to | Required |
| `version` | Semantic version number | `1.0.0` |
| `isEnabled` | Whether the plugin is active | `true` |
| `isOfficial` | Whether this is a Raven built-in plugin | `false` |
| `config` | Custom configuration object | `{}` |
| `code` | Plugin logic (for custom plugins) | `null` |
| `metadata` | Additional key-value metadata | `{}` |

## Plugin Context

Every plugin receives a context object with request metadata and content.

```json
{
  "metadata": {
    "orgId": "org_abc123",
    "keyId": "key_def456",
    "model": "gpt-4o",
    "provider": "openai"
  },
  "request": { "model": "gpt-4o", "messages": [...] },
  "response": { "choices": [...] }
}
```

## Plugin Results

A plugin returns a result indicating whether the request/response was modified or blocked.

```json
{
  "modified": true,
  "blocked": false,
  "response": { "choices": [...] },
  "blockReason": null
}
```

| Field | Description |
|-------|-------------|
| `modified` | Whether the plugin changed the request or response |
| `blocked` | Whether the plugin blocked the request entirely |
| `request` | Modified request body (for `pre_request` hooks) |
| `response` | Modified response body (for `post_response` hooks) |
| `blockReason` | Explanation if the request was blocked |

## Enabling and Disabling

Toggle plugins on or off without deleting them. Disabled plugins are skipped during request processing but their configuration is preserved.

## Plugin Execution Order

When multiple plugins are attached to the same hook, they execute sequentially. Each plugin receives the output of the previous plugin, creating a pipeline:

```
Request --> Plugin A (pre_request) --> Plugin B (pre_request) --> Provider
Provider --> Plugin C (post_response) --> Plugin D (post_response) --> Response
```

If any plugin sets `blocked: true`, the pipeline stops immediately and the request is rejected.
