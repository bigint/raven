---
title: "Caching"
description: "Reduce costs and latency with exact-match and semantic response caching."
---

Raven's caching layer intercepts requests and serves cached responses for identical or similar queries, reducing both latency and cost. It supports exact-match caching via Redis and semantic caching using trigram fingerprinting with Jaccard similarity.

## How It Works

```
Request --> Cache Check --> Hit? --> Return cached response (fast, free)
                       --> Miss? --> Provider --> Cache response --> Return
```

1. When a request arrives, Raven extracts the user messages and normalizes them
2. For exact matches, a hash lookup finds identical previous queries
3. For semantic matches, a trigram fingerprint is compared against recent cached entries
4. If a match is found above the similarity threshold, the cached response is returned
5. If no match, the request goes to the provider and the response is cached for future use

## Exact-Match Caching

Exact-match caching uses a normalized hash of the user messages. Two requests with identical content (after lowercasing and whitespace normalization) will produce the same cache key.

This is fast and deterministic -- perfect for:
- Identical repeated queries
- Batch processing of the same prompt
- FAQ-style interactions

## Semantic Caching

Semantic caching catches **similar** (but not identical) queries. It uses trigram fingerprinting with Jaccard similarity -- no embedding model required.

### How Semantic Matching Works

<Steps>
  <Step title="Text Normalization">
    User messages are lowercased, punctuation is removed, and whitespace is collapsed.
  </Step>
  <Step title="Trigram Fingerprinting">
    The normalized text is split into word-level trigrams (3-word sequences). These form a fingerprint set.
  </Step>
  <Step title="Jaccard Similarity">
    The fingerprint is compared against cached entries using Jaccard similarity: `intersection / union` of the two trigram sets.
  </Step>
  <Step title="Threshold Check">
    If the similarity score meets or exceeds the threshold (default: 0.92), the cached response is returned.
  </Step>
</Steps>

### Example

These two queries would likely match with a high similarity score:

```
Query 1: "What is the capital city of France?"
Query 2: "What is the capital of France?"
```

The trigram fingerprints would share most of their n-grams, producing a Jaccard similarity well above 0.92.

## Cache Configuration

| Setting | Description | Default |
|---------|-------------|---------|
| `enabled` | Toggle caching on/off | `true` |
| `similarityThreshold` | Minimum Jaccard similarity for a cache hit (0.0 - 1.0) | `0.92` |
| `ttl` | Time-to-live for cached entries in seconds | `86400` (24 hours) |
| `maxEntries` | Maximum cached entries per org per model | `100000` |

<Info>
  A similarity threshold of 0.92 is aggressive -- it only matches very similar queries. Lower the threshold (e.g., 0.80) to catch more matches at the risk of returning less relevant cached responses.
</Info>

### Minimum Query Length

Queries shorter than 10 characters after normalization are not cached. This prevents caching trivial or ambiguous prompts like "hi" or "help".

## Per-Organization Isolation

Cache entries are isolated per organization and per model. Organization A's cache is never visible to Organization B, and a cached GPT-4o response will not be served for a Claude request.

Cache key structure: `sem-cache:{orgId}:{model}:{entryId}`

## Cache Analytics

Monitor cache performance in the analytics dashboard:

| Metric | Description |
|--------|-------------|
| **Hit rate** | Percentage of requests served from cache |
| **Hit count** | Per-entry hit counter (most popular cached responses) |
| **Cost savings** | Estimated spend avoided by cache hits |
| **Latency improvement** | Average response time reduction |

## When Caching Works Best

<Tabs>
  <Tab title="Good Candidates">
    - **Repeated queries** -- FAQs, common questions, support chatbots
    - **Classification tasks** -- Sentiment analysis, categorization
    - **Extraction tasks** -- Parsing structured data from similar inputs
    - **Batch processing** -- Running the same prompt across similar documents
  </Tab>
  <Tab title="Poor Candidates">
    - **Creative generation** -- Where unique output is desired
    - **Real-time data** -- Queries that depend on current information
    - **Long conversations** -- Multi-turn chats with unique context
    - **Personalized responses** -- Outputs that should vary per user
  </Tab>
</Tabs>

## Flushing the Cache

To clear cached entries, use the dashboard or API. Cache entries automatically expire after the configured TTL.

<Note>
  Cache entries include a hit counter that increments each time the entry is served. High-hit entries indicate opportunities for prompt standardization or dedicated FAQ handling.
</Note>

## Cache Entry Structure

Each cached entry stores:

```json
{
  "response": "The capital of France is Paris.",
  "model": "gpt-4o",
  "fingerprint": ["what is the", "is the capital", "the capital of", "capital of france"],
  "createdAt": 1710489600000,
  "hitCount": 42
}
```
