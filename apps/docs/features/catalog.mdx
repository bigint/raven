The AI Catalog is an organization-wide marketplace for discovering, sharing, and governing AI resources. It provides an approval workflow so that models, agents, MCP servers, prompt templates, and guardrail policies can be vetted before they are used in production.

## Item Types

The catalog supports five resource types.

| Type | Description | Example |
|------|-------------|---------|
| `model` | Curated model configurations with recommended settings | GPT-4o with specific temperature and system prompt |
| `agent` | Pre-configured agent identity definitions | Customer support bot with scoped capabilities |
| `mcp_server` | Published MCP server connections | GitHub tools server, database query server |
| `prompt_template` | Shared prompt templates | Invoice extraction prompt, email drafting template |
| `guardrail_policy` | Reusable guardrail and policy configurations | PII detection policy for healthcare use |

## Approval Workflow

Every catalog item goes through an approval workflow before it becomes available.

| Status | Description |
|--------|-------------|
| `pending_approval` | Item has been submitted and is waiting for review |
| `approved` | Item has been reviewed and approved for use |
| `rejected` | Item was reviewed and not approved |
| `deprecated` | Item was previously approved but is no longer recommended |

<Steps>
  <Step title="Submit an Item">
    A team member submits a resource to the catalog with a name, description, and configuration.
  </Step>
  <Step title="Review">
    An admin or designated reviewer evaluates the item for quality, security, and compliance.
  </Step>
  <Step title="Approve or Reject">
    The reviewer approves the item (making it available to the organization) or rejects it with feedback.
  </Step>
  <Step title="Publish">
    Approved items appear in the catalog for all team members to discover and use.
  </Step>
</Steps>

### Submitting an Item via API

```bash
curl -X POST https://api.raven.dev/v1/catalog \
  -H "Authorization: Bearer rk_live_abc123" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Invoice Extraction Prompt",
    "type": "prompt_template",
    "description": "Extracts structured data from invoice PDFs with 95%+ accuracy",
    "version": "1.0.0",
    "tags": ["finance", "extraction", "production-ready"],
    "config": {
      "systemPrompt": "You are an invoice parser...",
      "model": "gpt-4o",
      "temperature": 0.1
    }
  }'
```

## Filtering and Search

Browse the catalog using multiple filters:

| Filter | Description |
|--------|-------------|
| **Type** | Filter by resource type (model, agent, mcp_server, etc.) |
| **Status** | Filter by approval status |
| **Tags** | Search by tags |
| **Search** | Full-text search across name and description |

## Catalog Item Details

Each catalog item tracks:

| Field | Description |
|-------|-------------|
| `name` | Item display name |
| `type` | Resource type |
| `description` | What the item does |
| `version` | Semantic version number |
| `status` | Approval status |
| `tags` | Searchable tags |
| `config` | Item-specific configuration |
| `qualityScore` | Quality rating (0-100) |
| `costEstimate` | Estimated cost per use |
| `usageCount` | How many times the item has been used |
| `createdBy` | Who submitted the item |
| `approvedBy` | Who approved the item |
| `approvedAt` | When it was approved |

## Usage Tracking

Each catalog item has a `usageCount` that tracks how many times it has been used. This helps identify:

- **Popular items** -- Most-used resources across the org
- **Underutilized items** -- Resources that may need better documentation or promotion
- **Deprecated alternatives** -- Old items that should be migrated away from

## Quality Scores

Items can be assigned a quality score (0-100) based on evaluations, user feedback, or manual review. Use quality scores to:

- Surface the best resources in search results
- Set minimum quality thresholds for production use
- Track quality improvements across versions

## Cost Estimates

Each item can include a cost estimate to help teams make budget-conscious decisions. For model configurations, this reflects the estimated per-request cost. For agents, it reflects the estimated cost per interaction.
