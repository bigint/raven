---
title: "Data Residency"
description: "Enforce data residency requirements by controlling which provider regions handle your requests."
---

Data residency controls let you restrict which geographic regions can process your AI requests. This is essential for compliance with regulations like GDPR, which require that personal data remains within specific jurisdictions.

## How It Works

Raven maintains a mapping of each provider to the regions where they process data. When you configure data residency rules, Raven filters providers to ensure requests only go to providers operating in your allowed regions.

## Provider Region Mappings

Raven tracks region information for each supported provider:

| Provider | Regions | Notes |
|----------|---------|-------|
| **OpenAI** | `us` | US-based processing |
| **Anthropic** | `us` | US-based processing |
| **Mistral AI** | `eu` | EU-based (France) |

### Region Codes

| Code | Region |
|------|--------|
| `us` | United States |
| `eu` | European Union |
| `any` | No restriction (all regions allowed) |

## Configuring Allowed Regions

Set your allowed regions in the organization settings or via a policy.

<Steps>
  <Step title="Navigate to Settings">
    Go to your organization **Settings** in the dashboard.
  </Step>
  <Step title="Select Data Residency">
    Open the **Data Residency** section.
  </Step>
  <Step title="Choose Allowed Regions">
    Select one or more regions where your data is permitted to be processed.
  </Step>
  <Step title="Enable Strict Enforcement">
    Optionally toggle strict enforcement mode for unknown providers.
  </Step>
</Steps>

## Strict Enforcement Mode

When strict enforcement is **enabled**:

- Providers with **unknown** region mappings are **blocked**
- Only providers with explicitly mapped regions that match your allowlist are permitted
- This is the safest option for regulated environments

When strict enforcement is **disabled** (default):

- Providers with unknown region mappings are **allowed**
- This is more permissive but ensures new providers work without configuration updates

<Warning>
  Enabling strict enforcement may block requests to newly added providers that Raven has not yet mapped to a region. Monitor your request logs after enabling this setting.
</Warning>

## Filtering Providers by Region

When data residency is configured, Raven automatically filters the available providers during request routing:

```
Configured providers:     [openai, anthropic, mistralai]
Allowed regions:          [us, eu]

After data residency filtering:
  openai       --> us       Allowed
  anthropic    --> us       Allowed
  mistralai    --> eu       Allowed
```

This filtering happens transparently. Your application does not need to be aware of which providers are excluded -- Raven handles routing to compliant providers automatically.

## GDPR Article 44 Compliance

For organizations subject to GDPR, data residency controls help satisfy **Article 44** requirements for cross-border data transfers:

- Restrict processing to `eu` region only to keep data within the EU
- Use Mistral AI (France-based) as your primary provider for EU data residency
- Enable strict enforcement to prevent accidental routing to non-EU regions

### Example: EU-Only Configuration

```json
{
  "allowedRegions": ["eu"],
  "enforceStrictly": true
}
```

With this configuration:

| Provider | Region | Result |
|----------|--------|--------|
| Mistral AI | `eu` | Allowed |
| OpenAI | `us` | Blocked |
| Anthropic | `us` | Blocked |

### Example: US Only

```json
{
  "allowedRegions": ["us"],
  "enforceStrictly": false
}
```

## Combining with Routing Rules

Data residency works alongside [routing rules](/features/routing-rules). When both are configured:

1. Data residency filtering runs first, removing non-compliant providers
2. Routing rules then operate on the remaining compliant providers
3. If no compliant providers remain, the request is rejected with an error

## BYOK and Data Residency

With [Bring Your Own Key](/api-reference/authentication), you can send your own provider API key via the `X-Provider-Key` header. Data residency checks still apply -- even with BYOK, the provider must operate in an allowed region. This prevents circumventing data residency controls.

## Error Responses

When a request is blocked by data residency rules:

```json
{
  "error": {
    "message": "Provider openai operates in us but only eu allowed",
    "code": "DATA_RESIDENCY_VIOLATION"
  }
}
```
